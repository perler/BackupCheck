# BackupCheck

Backup monitoring for Macrium Reflect with [healthchecks.io](https://healthchecks.io) alerting.

Scans backup repositories for recent `.mrimg` files, reports per-machine health to healthchecks.io, and auto-configures check settings based on device type. Runs as a Windows Scheduled Task.

## Features

- Monitors multiple backup repositories (local or UNC network shares)
- Auto-detects Macrium Reflect repositories via Site Manager
- Auto-provisions healthchecks.io checks on first ping
- Auto-configures check period/grace based on device naming convention
- Self-updating from GitHub releases
- HC API caching to minimize Management API calls
- Structured logging with 7-day rotation
- Meta-monitoring (monitors the monitor itself)
- Skips machines with active backups in progress
- Detects corrupted backup files (`.error_loading`)

## Device Type Settings

| Device Prefix | Tag | Period | Grace |
|---|---|---|---|
| `wks*` (workstation) | `wks` | 4 days | 6 hours |
| `nb*` (notebook) | `nb` | 8 days | 6 hours |
| `srv*` (server) | `srv` | 1 day | 18 hours |

## Quick Start

### 1. Set up healthchecks.io

1. Create a project at [healthchecks.io](https://healthchecks.io)
2. Get your **Ping Key** and **API Key** from Project Settings > API Access
3. Configure alerting integrations (email, Slack, etc.)

### 2. Install

Download the latest release and run the installer as Administrator:

```powershell
.\Install-BackupMonitor.ps1
```

The installer prompts for:
- **Company ID**: Short prefix for check names (e.g., `ACME` creates checks like `acme-wks001`)
- **Ping Key / API Key**: From healthchecks.io
- **Repository credentials**: For NAS/share access
- **Task credentials**: Windows/AD account for the scheduled task

### 3. Verify

1. Check Task Scheduler for the `BackupMonitor` task
2. Run manually: `.\Monitor-Backups.ps1`
3. Check your healthchecks.io dashboard

## Configuration

### config.json

Generated by the installer:

| Setting | Description | Default |
|---|---|---|
| `configVersion` | Config format version | `2` |
| `companyId` | Prefix for check names | (set during install) |
| `repositories` | Backup repository paths | (auto-detected) |
| `backupMaxAgeHours` | Max backup age in hours | `24` |
| `backupFilePattern` | File pattern for backups | `*.mrimg` |
| `skipIfRunning` | Skip machines with active backups | `true` |
| `healthchecksBaseUrl` | Ping URL | `https://hc-ping.com` |
| `autoDetectRepositories` | Auto-detect repos on each run | `true` |
| `updateUrl` | Custom URL for update checks | (GitHub raw URL) |

### .env

```
HC_PING_KEY=your-ping-key
HC_API_KEY=your-api-key
REPO_USERNAME=nas002\backupuser
REPO_PASSWORD=your-password
```

See [.env.example](.env.example) for all available fields.

## How It Works

1. **Log rotation**: Cleans entries older than 7 days from `backupcheck.log`
2. **Self-update check**: Fetches `latest.json` from GitHub (once per 24h), downloads and applies updates if available
3. **Repository scanning**: For each repository, scans subdirectories (one per machine)
4. **Health check**: For each machine, checks for recent `.mrimg` files within the configured threshold
5. **HC ping**: Sends success/fail ping with version-tagged status message
6. **HC configuration**: Auto-configures check period, grace, and tags (cached to reduce API calls)
7. **Meta-monitoring**: Pings a `{companyId}-monitor-health` check after each run

### Check Naming

Pattern: `{companyId}-{machineName}` (lowercased)

Examples: `acme-wks001`, `acme-srv003`, `acme-nb002`

### Self-Update

The monitor checks GitHub for updates every 24 hours:
- Fetches `latest.json` from the repo
- Compares version strings
- Downloads release zip, verifies SHA256 checksums
- Backs up current files (`.bak`), extracts new ones
- Re-launches with `-SkipUpdateCheck` to avoid loops

If the update fails at any step, the current version continues running.

## Offline Machine Handling

For workstations and notebooks that are legitimately offline (weekends, holidays, travel), the companion script `pause-offline-checks.py` runs on the coordinator server and:

- Queries Atera RMM for agent online/offline status
- Mutes notification channels for offline machines (prevents false alerts)
- Restores channels when machines come back online

This will be replaced by the coordinator API in v2.1.

## Troubleshooting

### Checks not appearing
- Verify ping key in `.env`
- Run `Monitor-Backups.ps1` manually and check console + `backupcheck.log`
- Ensure TLS 1.2 connectivity to healthchecks.io

### Access denied to repositories
- Verify `REPO_USERNAME`/`REPO_PASSWORD` in `.env`
- Check the scheduled task user has network access

### False "down" alerts
- Adjust `backupMaxAgeHours` for your backup schedule
- Check device type settings match your check periods
- For offline machines, ensure `pause-offline-checks.py` is running

### Updates not applying
- Check `backupcheck.log` for update errors
- Verify internet access to GitHub from the server
- Delete `.last-update-check` to force an immediate check
